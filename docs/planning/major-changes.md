# Major Changes

Append-only record of major project changes. Add entries in the order performed, with the oldest entry at the top and the newest entry at the bottom.

Each entry must start with the date, state what was added or changed, and explain why it was added. Keep entries concise and factual.

## Entries

- 2026-04-27 - **Added PostgreSQL pool observability**: Added periodic `pgxpool` stats reporting and New Relic custom metrics for connection pool pressure and acquire wait time. This was added to make database saturation visible in staging and production, so pool sizing and backend behavior can be tuned from measurements.
- 2026-04-27 - **Added frontend Web Vitals reporting to New Relic**: Added a client-side Web Vitals reporter that forwards metrics through the existing New Relic browser agent. This was added to connect real frontend performance signals to the existing observability pipeline.
- 2026-04-27 - **Enabled Next.js Cache Components and Partial Prerendering**: Enabled Cache Components in the Next.js app and moved cacheable public station reads behind server-only cache wrappers and Suspense boundaries. This was added to improve prerendering and response performance while keeping request-scoped data, such as CSP nonces and locale params, compatible with the Next.js runtime model.
- 2026-04-27 - **Reorganized the documentation tree by audience**: Moved flat top-level docs into architecture, operations, planning, editorial, and security sections, added a docs index, and renamed misleading files such as the pseudo-YAML tag reference and the station entry guide. This was added to make the documentation easier to navigate and to separate runbooks, technical reference, editorial guidance, and project backlogs.
- 2026-04-27 - **Rebuilt the admin overview around system ownership metrics**: Split the station-health overview onto the editor surface and added a new admin overview contract for system status, user access, billing, content pipeline, and media storage metrics. This was added so admins get platform-level insight without duplicating editor station operations.
- 2026-04-27 - **Added admin diagnostic drill-down pages**: Added separate admin API, database, and jobs diagnostics pages backed by dedicated admin endpoints and store-owned database/job summaries. This was added so admins can inspect detailed runtime, pool, migration, and background-worker freshness without overloading the admin overview.
- 2026-04-27 - **Added manual admin job triggers**: Added admin-only controls and endpoints to trigger station sync and stream re-probe jobs with duplicate-run guards. This was added so operators can refresh system state on demand from the jobs diagnostics page without shell access.
- 2026-04-27 - **Scoped recurring stream probes to approved stations**: Changed scheduled and admin-triggered stream re-probe jobs plus jobs diagnostics to operate on active streams for approved stations only. This was added to avoid spending recurring background-worker resources on pending or rejected catalog candidates while preserving explicit editor/admin stream probes during curation.
- 2026-04-27 - **Added stream probe scheduling and typed failure codes**: Added `next_probe_at` and `last_probe_error_code` to stream rows, changed recurring probes to consume only due approved streams, and documented the probe evidence pipeline in detail. This was added to make stream maintenance budget-aware and to replace free-text probe failures as the scheduling contract.
- 2026-04-27 - **Expanded metadata standards support**: Added first-class metadata routing values for ID3, Vorbis comments, HLS, DASH, and EPG, added server-side ID3/Vorbis extraction, DASH/OGG stream classification, and a global metadata poller fetch governor. This was added to align metadata discovery and client/server routing with common streaming metadata standards while protecting upstream fetch capacity.
- 2026-04-27 - **Added bulk metadata coverage trigger**: Added an admin jobs action that fetches now-playing metadata for all approved metadata-enabled streams through the existing metadata poller pressure controls. This was added so operators can test metadata coverage from the jobs diagnostics page without opening each station individually.
- 2026-04-27 - **Allowed client metadata stream fetches in CSP**: Expanded the frontend `connect-src` policy to permit HTTPS stream origins for browser-resolved metadata. This was added so CORS-capable streams can remain on the client resolver path and avoid unnecessary backend metadata pressure.
- 2026-04-27 - **Added typed metadata delivery plans**: Added a backend-owned `metadata_plan` contract to stream responses and used it to align runtime metadata delivery and low-pressure client fallback behavior. This was added to make resolver behavior more explicit, easier to extend, and less prone to regressions across client/server metadata paths.
- 2026-04-27 - **Deduplicated client metadata polling across tabs**: Added per-stream client metadata lease ownership and shared now-playing snapshots in browser storage so only one tab polls CORS-capable streams at a time while follower tabs render the shared result. This was added to reduce browser-side upstream pressure without moving client-capable streams back onto the backend path.
- 2026-04-27 - **Added supplemental metadata providers**: Added stream-level provider configuration and backend provider implementations for NPR Composer and NTS Live, with editor controls for configuring provider-specific station identifiers. This was added to cover streams whose native ICY/ID3/HLS metadata is absent while keeping provider logic in code and station-specific values in editorial data.
- 2026-04-28 - **Hardened no-metadata delivery behavior**: Changed confirmed backend metadata misses to persist `no_metadata`, disable the server resolver, stop active server poll loops, expose metadata error snapshots in stream responses, and make the player use `metadata_plan.delivery` as its runtime contract. This was added so playable streams without discoverable metadata do not show metadata badges or create backend metadata pressure.
- 2026-04-28 - **Added pending media asset expiry and backend cleanup**: Added `expires_at` column to `media_assets` (migration 040), set to 20 minutes on `CreatePending` and cleared on `MarkReady`/`MarkRejected`. Added `DeleteExpiredPending` store method and a `MediaCleaner` background goroutine that evicts expired rows and their uploaded blobs every 10 minutes. Updated the CI `cleanup.yml` to exclude pending assets (`status != 'pending'`) since the backend now owns that lifecycle. This was added to prevent abandoned upload-intent rows from accumulating indefinitely when clients fail between `upload-intent` and `complete`.
- 2026-04-28 - **Added admin media & storage diagnostics page**: Added `GET /admin/diagnostics/media` endpoint backed by a new `AdminDetailedSummary` store query, and a corresponding frontend page at `/admin/media` wired into the admin nav. The page surfaces pipeline counts (total/ready/pending/rejected, success and rejection rates), per-kind breakdowns (avatars vs station icons with storage), and storage footprint (total bytes, average asset size, content-hash coverage). This was added to replace the MVP media card on the admin overview with a dedicated, actionable diagnostics surface.
- 2026-04-28 - **Hardened media expiry cleanup claims and indexing**: Changed pending media expiry cleanup to run immediately on backend startup, claim expired pending rows before blob deletion, retry cleanup by releasing failed claims, and block claimed uploads from being marked ready or rejected. Added a partial `media_assets` expiry index for unclaimed pending rows. This was added to prevent orphaned blobs, keep cleanup retryable, and ensure the recurring expiry worker stays cheap under production traffic.
- 2026-04-28 - **Restored client metadata routing after server misses**: Changed the metadata poller to re-check browser-readable metadata support before writing `resolver = none` after a `no_metadata` fetch, and updated frontend metadata surfaces to trust client delivery plans over stale backend miss snapshots. This was added to keep CORS-capable streams such as SomaFM on the client metadata path instead of silently downgrading them to disabled metadata.
- 2026-04-28 - **Split media cleanup claims into migration 041**: Restored migration 040 to the originally shipped `expires_at` column change and added migration 041 for `cleanup_claimed_at` plus the pending-expiry partial index. This was added because environments that had already applied version 040 would not rerun an edited migration file, which left the new cleaner code ahead of the schema.
- 2026-04-28 - **Stopped editor metadata saves from preserving stale disabled state**: Changed editor stream rebuilds and station-detail saves to treat metadata as enabled by default again, taught stream responses and now-playing routing to recover from known-good metadata hints/providers, and added migration 042 to re-enable recoverable rows already stuck in `metadata_enabled = false`. This was added so CORS-capable streams such as SomaFM stop getting trapped behind legacy disabled flags after the metadata routing refactor.
- 2026-04-28 - **Added on-read metadata route classification for editor station detail**: Changed `GET /editor/stations/:id` to run a lightweight metadata route probe for active streams that are enabled but still unclassified, then persist the resolver check result before building the response. This was added so the internal editor view can self-heal metadata diagnostics for streams like SomaFM without bringing back manual probe controls.
- 2026-04-28 - **Centralized metadata route classification behind a shared router**: Added a single backend metadata router service and moved recurring probes, poller `no_metadata` fallback, editor station-detail cold reads, and explicit admin metadata probes onto that contract. This was added to stop metadata delivery decisions from drifting across code paths and to make SomaFM-style client-readable streams classify consistently everywhere.
- 2026-04-28 - **Added structured backend platform logging foundation**: Added a shared `slog` logger constructor, request correlation middleware, and event-shaped logs for now-playing routing, metadata fetches, and stream probe workflows. This was added to establish a local/staging/production logging contract before centralizing logs in the platform observability stack.
- 2026-04-28 - **Added backend OpenTelemetry tracing foundation**: Added OpenTelemetry SDK/exporters, Gin request instrumentation, trace resource configuration, and trace IDs in structured request logs. This was added so staging and production can export correlated traces through OTLP while local development remains opt-in.
- 2026-04-28 - **Added Azure Monitor Application Insights telemetry wiring**: Added a workspace-based Application Insights resource, enabled the Azure Container Apps managed OpenTelemetry agent, and configured the backend to export OTLP traces through the environment agent. This was added to centralize staging and production trace/log correlation in Azure Monitor without running a custom collector.
- 2026-04-28 - **Removed backend New Relic APM**: Removed the backend New Relic Go agent, Gin middleware, runtime config, and infrastructure variables now that backend traces and logs route through Azure Monitor Application Insights. This was added to avoid duplicate backend instrumentation and keep the platform observability path Azure-native.
- 2026-04-28 - **Added backend outbound dependency tracing**: Added safe OpenTelemetry client spans for Radio Browser, stream probes, metadata providers, metadata fetchers, HLS ID3 probes, and media blob storage, plus Azure Monitor workbook and alert query starting points. This was added to expose upstream pressure and latency without recording full stream or blob URLs.
- 2026-04-28 - **Reduced development GitHub Actions usage**: Disabled duplicate `push main` CI, removed PR Docker validation and coverage artifact uploads, moved heavy security scans to scheduled/manual runs, added local VS Code validation tasks, and recorded a launch-readiness item to restore the full gate set. This was added to cut Actions billing sharply during development while keeping essential PR checks and a documented path back to full automation before launch.
- 2026-04-28 - **Switched deployments to manual dispatch only**: Removed the automatic `push main` trigger from the deploy workflow and simplified the workflow to run only from `workflow_dispatch`. This was added to stop every merge from consuming deployment minutes during development while preserving explicit staging and production release control.
- 2026-04-28 - **Tightened frontend CSP nonce delivery and Safari script allowlisting**: Passed the request nonce into the locale theme bootstrap, replaced the broad HTTPS script fallback with explicit external script origins, and stopped forwarding the full request header set through middleware overrides. This was added so Safari/login rendering remains compatible with strict CSP without widening script execution or inflating per-request header overhead.
- 2026-04-28 - **Moved public station caching onto the fetch layer**: Removed `use cache` wrappers around public station feed/detail/filter loaders and applied revalidation directly to the underlying public fetch calls. This was added to keep prerender-safe caching on the data layer and avoid Next.js cache-fill timeouts when route inputs such as locale params or search params flow into those reads.
- 2026-04-28 - **Bounded public station fetch time during prerender**: Added a 5-second timeout to public station feed/detail/filter fetches. This was added so frontend prerender falls back to the existing empty/error states when the API is unreachable during image builds instead of hanging until Next.js aborts the page after 60 seconds.
- 2026-04-28 - **Moved station-backed protected routes to request-time rendering**: Marked the CarPlay, curated, explore, and curated-detail routes as dynamic so Next.js no longer prerenders them during the frontend image build. This was added to remove hard build-time dependencies on live station API availability while keeping the pages free to fetch fresh public catalog data at request time.
- 2026-04-28 - **Allowlisted Next.js prerender timing bootstrap by hash**: Added the SHA-256 hash for Next.js’s tiny inline `requestAnimationFrame` prerender timing script to the frontend CSP. This was added so prerendered pages keep a strict CSP without broad inline script allowance while avoiding the remaining browser-reported `script-src-elem` violation.
- 2026-04-28 - **Removed `strict-dynamic` from the frontend CSP**: Kept nonce/hash protection for inline scripts and explicit host allowlists for trusted external scripts, but removed `strict-dynamic` because prerendered Next.js pages emit static same-origin chunk tags without per-request nonces. This was added so static and partially prerendered pages can load their `/_next/static` framework scripts under CSP instead of being blocked in browsers that correctly ignore host allowlists when `strict-dynamic` is present.
- 2026-04-28 - **Corrected the CSP hash for Next.js’s inline timing script**: Updated the `script-src` SHA-256 entry for the prerendered `requestAnimationFrame(function(){$RT=performance.now()});` bootstrap after verifying the exact built HTML output. This was added because the previous hash did not match the real script body and left one `script-src-elem` violation on prerendered pages such as login.
- 2026-04-28 - **Unified OAuth login return-path handling**: Standardized frontend auth redirects on `callbackUrl`, taught the login page to honor the legacy `redirect` query as a fallback, and surfaced NextAuth OAuth error codes on the login screen. This was added so GitHub/Google login flows return to the intended page consistently and provider failures are visible instead of looking like a silent broken login.
- 2026-04-28 - **Extended frontend OAuth exchange timeout for cold-started staging auth**: Increased the server-side timeout for the frontend-to-backend `/auth/oauth` exchange to 30 seconds. This was added because staging runs the backend with zero minimum replicas, and GitHub login could fail during backend cold start before the default 10-second timeout elapsed.
- 2026-04-28 - **Moved frontend server-side auth calls onto the backend ingress FQDN**: Changed the frontend container’s runtime `API_URL` to use the backend Container App ingress hostname instead of the public custom domain, while keeping the public browser API URL as a separate deploy variable. This was added so NextAuth server-side exchanges such as `/auth/oauth` avoid public-domain routing quirks and talk directly to the backend app endpoint inside the platform path.
- 2026-04-28 - **Aligned backend OpenTelemetry semantic conventions with the SDK schema**: Updated the backend tracing bootstrap to use `semconv/v1.40.0` so its resource attributes match the schema URL emitted by the current OpenTelemetry SDK defaults. This was added to stop backend startup from failing on resource-merge schema conflicts in staging.
- 2026-04-28 - **Unified stream diagnostics persistence behind a shared backend contract**: Added a shared `StreamDiagnosticsUpdate` store contract for quality, loudness, and metadata evidence, and moved editor probes, recurring approved re-probes, metadata fetch coverage, and editor metadata cold-healing onto that write path. This was added to stop probe surfaces from drifting on which fields they update and to reduce recurring regressions between manual editor actions, background jobs, and metadata polling.
- 2026-04-28 - **Stopped pending editor probes from writing public now-playing state**: Restricted editor/admin metadata probes so only approved stations persist snapshots into `stream_now_playing`, while pending and rejected stations still keep stream-level diagnostics and routing evidence. This was added because shared now-playing snapshots are part of the public/listener-facing contract and pending catalog candidates should not accumulate public metadata state during editorial curation.
- 2026-04-28 - **Reset stream metadata onto an explicit end-to-end contract**: Replaced `metadata_enabled` with `metadata_mode` (`auto` or `off`), made `metadata_resolver` explicit (`unknown`, `client`, `server`, `none`), removed editor read-side metadata healing, stopped startup station sync and stream re-probe from mutating metadata immediately on boot, and aligned backend/public/editor/frontend contracts around those explicit states. This was added to eliminate heuristic metadata recovery, make approved-only public metadata behavior predictable, and keep all metadata surfaces reflecting one persisted contract instead of whichever probe path happened to run first.
